## 나는 개발자가 아니다

정확히 말하면, 회사에서 개발을 하지 않는다. 기획이 본업이다. 코드를 읽을 줄은 알지만, 빈 파일에서 앱을 만들어본 적은 없었다.

그런데 지금 Flutter 앱 3개를 동시에 만들고 있다. GentleDo(할일 관리), GentleFast(간헐적 단식), GentleStudy(학습 플래너). GentleDo는 TestFlight까지 갔고, 나머지 둘도 MVP가 나왔다.

비결은 Claude Code다.

## 바이브코딩이란

"바이브코딩"이라는 말이 있다. AI에게 자연어로 지시해서 코드를 생성하는 방식. 나는 이걸 매일 한다.

실제 작업 흐름은 이렇다.

1. **PROJECT_BRIEF.md** 에 만들고 싶은 기능을 한국어로 쓴다
2. **CLAUDE.md** 에 코딩 규칙을 정의해둔다 (Riverpod 쓸 것, 빨간색 금지, ARB로 다국어 등)
3. Claude Code에게 "오늘 할일 탭의 타임라인 뷰를 만들어줘"라고 말한다
4. Claude가 코드를 생성하고, 나는 시뮬레이터에서 확인한다
5. "이 부분 간격이 좀 좁아" "다크모드에서 안 보여" 같은 피드백을 주고받는다

한 마디로, **설계는 내가 하고, 구현은 Claude가 한다.**

## 실제로 만든 것들

GentleDo 기준으로, Claude Code와 함께 만든 기능 목록이다.

- 에너지 체크인 시스템 (Low/Mid/High)
- 모멘텀 점수 (0-100, 감쇠 알고리즘)
- 24시간 타임라인 뷰
- 주간 타임블록 뷰
- 집중 모드 (타이머 + 단일 태스크)
- 새 출발 (밀린 태스크 일괄 정리)
- 반복 태스크 (매일/매주/매월)
- 서브태스크
- 드래그 앤 드롭 시간 배치
- 미루기 로그 (5가지 사유 + 패턴 분석)
- 온보딩 8페이지
- 알림 5종
- 데이터 내보내기/가져오기
- 다크 모드
- 한국어/영어 다국어
- 홈 위젯 (iOS WidgetKit + Android AppWidget)
- 데스크톱 지원 (macOS, Windows)

테스트 케이스 158개. 전부 통과한다.

이걸 혼자서, 개발 경험 없이 만들었다고 하면 아무도 안 믿는다. 사실 "혼자"는 아니다. Claude가 짝 프로그래머였다.

## 잘 되는 것

**CLAUDE.md의 힘.** 프로젝트 시작할 때 규칙 문서를 꼼꼼히 쓰면, Claude가 그 규칙을 계속 지킨다. "빨간색 쓰지 마" "Riverpod만 써" "UI 문자열 하드코딩하지 마" — 이런 걸 한 번 적어두면 수백 개 파일에 걸쳐 일관성을 유지한다.

**반복 작업이 빠르다.** 한국어 ARB 파일에 문자열 50개를 추가하고, 영어 번역까지 만들고, 코드에서 참조하는 작업 — 사람이 하면 2시간, Claude는 3분이다.

**테스트를 잘 쓴다.** "이 provider에 대한 테스트를 작성해줘"라고 하면 정상 케이스, 경계 케이스, 에러 케이스를 다 커버하는 테스트를 만든다. 내가 생각 못한 edge case도 잡아준다.

## 안 되는 것

**복잡한 UI 레이아웃에서 삽질한다.** "타임라인 뷰에서 드래그하면 시간이 바뀌는데, 동시에 스크롤도 되어야 해" 같은 요구사항은 한 번에 안 된다. 3~4번 수정 요청을 거치면서 점점 맞춰간다.

**전체 아키텍처를 미리 잡아야 한다.** Claude는 시키는 것은 잘 하지만, "이 앱의 전체 구조를 어떻게 잡으면 좋을까"에 대해서는 내가 결정해야 한다. 설계 없이 기능만 요청하면 나중에 전부 엎어야 한다. 실제로 두 번 엎었다.

**디버깅은 여전히 어렵다.** 에러가 나면 Claude에게 에러 메시지를 보여주고 고치라고 한다. 대부분 고치지만, 가끔 원인을 잘못 짚고 엉뚱한 곳을 수정하는 경우가 있다. 그때는 내가 에러 메시지를 읽고 "이 파일의 이 줄이 문제인 것 같아"라고 힌트를 줘야 한다.

## CLAUDE.md 작성 팁

바이브코딩을 시작하려는 사람에게 하나만 조언한다면: **CLAUDE.md를 먼저 쓰세요.**

내가 쓰는 CLAUDE.md에는 이런 것들이 들어있다.

- 프로젝트 개요 (한 줄)
- 기술 스택 테이블
- 코드 컨벤션 (네이밍, 임포트 순서, 상태 관리 방식)
- 절대 하지 말 것 목록 (빨간색 금지, 하드코딩 금지, 특정 패키지 금지)
- Phase별 범위 (지금 단계에서 뭘 하고, 뭘 하면 안 되는지)

이 문서가 있으면 Claude가 맥락을 이해한 상태로 일한다. 없으면 매번 처음부터 설명해야 한다.

## 현재 상태

| 앱 | 상태 | 테스트 |
|----|------|--------|
| GentleDo | TestFlight (베타 모집 중) | 158개 통과 |
| GentleFast | MVP 완성 (로컬) | 69개 통과 |
| GentleStudy | MVP 완성 (로컬) | 진행 중 |

3개 앱 모두 같은 디자인 시스템(No Red, Momentum, Energy Check-in)을 공유한다. 코드도 패턴을 공유한다. Claude Code가 첫 번째 앱에서 배운 패턴을 두 번째, 세 번째 앱에 적용한다.

개발자가 아니어도 앱은 만들 수 있다. 다만, 설계자는 되어야 한다. 무엇을 만들지, 왜 만드는지, 어떤 규칙으로 만들지 — 이건 아직 사람의 몫이다.
