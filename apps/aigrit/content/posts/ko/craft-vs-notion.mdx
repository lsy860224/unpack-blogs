**결론부터: Craft는 글쓰기에, Notion은 DB·협업에 강하다. 주 용도에 따라 선택하고, 양쪽을 역할 분리로 공존시키는 것도 가능하다.**

Craft와 Notion을 각 1년 이상 써본 비교다. 어느 한 쪽을 "무조건 낫다"고 말할 수 없다 — 하는 일이 다르기 때문이다. 나는 현재 둘 다 쓰고 있고, 그 역할 분리 방식까지 이 글에서 공개한다.

## 핵심 비교 테이블

| 항목 | Craft | Notion |
|---|---|---|
| **핵심 강점** | 글쓰기·디자인·속도 | DB·관계형 데이터·협업 |
| **에디터** | 블록 기반, Apple 네이티브 | 블록 기반, 웹 기반 |
| **속도** | 즉시 반응 (네이티브) | 로딩 1~2초 (웹) |
| **오프라인** | ✅ 완벽 | ⚠️ 제한적 |
| **복붙 호환** | ✅ 리치 텍스트 유지 | ⚠️ 마크다운 변환 |
| **DB 기능** | ❌ | ✅ 관계형 DB |
| **API** | ✅ (MCP 포함) | ✅ (강력한 API) |
| **협업** | 소규모 (5인 이하) | 대규모 (무제한) |
| **가격** | 무료~$5/월 | 무료~$10/월 |
| **플랫폼** | Apple 전용 | 모든 플랫폼 |

같은 "블록 기반 노트앱" 이지만 UI 철학이 다르다. **Craft는 "이 문단이 얼마나 좋은가"를 묻고, Notion은 "이 페이지가 전체에서 어디에 있는가"를 묻는다.** 해결하려는 문제가 다르니 선택 기준도 달라진다.

## Craft가 Notion보다 나은 점은?

실제 4가지 측면에서 체감 차이가 크다.

1. **글쓰기 경험.** Apple 네이티브 앱이라 타이핑 반응이 즉각적. Notion은 웹 기반이라 3,000단어 넘어가는 긴 문서에서 한 글자 입력과 화면 반영 사이 체감 지연이 생긴다. 하루에 1~2편 쓰는 입장에선 누적 피로도가 크다.
2. **복붙 품질.** Craft에서 복사한 내용은 네이버 스마트에디터·WordPress 구텐베르크에서 테이블·이미지·포맷이 유지된다. Notion은 마크다운으로 변환되면서 테이블이 깨진다. <Link href="/blog/craft-naver-workflow">Craft → 네이버 복붙 워크플로우</Link>에서 이 차이가 발행 워크플로우 전체를 바꾸는 이유를 자세히 다뤘다.
3. **오프라인.** 비행기·지하철에서도 완벽 작동. Notion은 캐시된 페이지만 열리고 편집은 동기화 시점에 충돌이 가끔 난다.
4. **디자인.** 깔끔한 문서 디자인이 기본값. Notion은 커스텀이 많지만 기본 상태는 정보밀도가 높아 "읽기"보다 "관리" 용도로 기운다.

## Notion이 Craft보다 나은 점은?

반대도 분명하다.

1. **관계형 DB.** 블로그 글 100개를 주제·발행일·조회수로 필터링하고 싶으면 Craft로는 불가능하다. <Link href="/blog/obsidian-seo-dashboard">Obsidian Dataview 대시보드</Link>로 대체 가능하지만 Notion 만큼 쉽지는 않다.
2. **협업.** 팀 위키로 Notion을 능가할 대안이 많지 않다. Craft는 5인 이내 협업까지 가능하지만 그 이상은 버거워진다.
3. **플랫폼 호환.** Android·Windows 동료와 공유해야 한다면 Notion 외에 선택지가 없다. Craft는 2026년 현재도 Apple 전용.
4. **Notion AI.** 워크스페이스 데이터를 직접 읽고 요약하는 기능. <Link href="/ko/blog/notion-ai-guide">Notion AI 완벽 활용법</Link>에서 다룬 "끊김 없이" 작동하는 경험은 Craft 에는 없다.

## 혼자 쓰면 어느 쪽이 나을까?

| 주 용도 | 추천 |
|---|---|
| 블로그 글쓰기 | Craft |
| 프로젝트 관리·DB | Notion |
| 일기·메모 | Craft (속도) |
| 팀 위키 | Notion |
| PKM (제텔카스텐) | 둘 다 아님 — Obsidian |

**나의 선택:** Craft = 글 초안 + 복붙 발행, Obsidian = SEO 기획·추적, Notion = 안 씀 (1인이라 관계형 DB 필요 없음). <Link href="/ko/blog/solo-developer-automation-stack">1인 개발자 자동화 스택</Link>의 "#8 Craft" 항목에서도 이 역할 분리가 왜 의미 있는지 자세히 정리했다.

## 전환 경험 — Notion → Craft 부분 이전

모든 것을 Notion에서 Craft로 옮길 필요는 없다. 나는 "**글쓰기**" 만 Craft로 옮겼다. Notion의 DB, 프로젝트 보드, 위키는 유지하거나 Obsidian 으로 분리.

**이전 과정 (실측 2시간):**

1. 글쓰기 관련 페이지만 선택 (약 40개)
2. Notion → 마크다운 내보내기 (Export as Markdown)
3. Craft에서 마크다운 임포트 (포맷 일부 깨짐 → 수동 정리)
4. 이후 새 글은 Craft 에서만 작성

**깨진 포맷 처리 패턴:**

- 토글 블록 → Craft 는 지원 X → 일반 헤딩으로 대체
- 데이터베이스 뷰 → Craft 는 지원 X → 일반 테이블로 평평화
- 임베드 위젯 → Craft 에서 깨짐 → 링크 텍스트로 치환

2시간 투자 후 뒤돌아보지 않았다. 단, 글 수가 100개 이하라 가능했던 전환이다. 300개 넘으면 자동화 스크립트 고려가 필요하다.

## API·통합 시각

| 항목 | Craft | Notion |
|---|---|---|
| 공개 API | ✅ | ✅ (더 성숙) |
| MCP 지원 | ✅ 네이티브 | ✅ 서드파티 |
| Webhook | ⚠️ 제한적 | ✅ 정식 |
| 자동화 도구 연결 | Make·Zapier 가능 | 무제한 통합 |

자동화 관점에선 Notion 이 한 수 위. <Link href="/ko/blog/apple-shortcuts-ai-automation">Apple 단축어 AI 자동화</Link> 에서 소개한 "음성 → Notion DB 저장" 같은 파이프라인은 Notion API 가 강력해서 가능한 것.

## 총정리

| 상황 | 선택 |
|---|---|
| Apple 사용자 + 글쓰기 중심 | **Craft** |
| 팀 협업 + DB 필요 | **Notion** |
| 혼자 + 최소 비용 | Craft 무료 (1,000 블록 내) |
| PKM 구축 | Obsidian (별도 도구) |
| 양쪽 다 | 역할 분리하면 공존 가능 |

<Link href="/ko/blog/claude-code-vs-cursor">Claude Code vs Cursor 비교</Link>에서도 같은 결론이었다 — "둘 중 하나"가 아니라 "역할 분리". 이 패턴은 1인 개발자의 도구 선택에서 반복되는 정답이다.

## 자주 묻는 질문 (FAQ)

### Craft 무료로 충분한가요?

1,000 블록 제한이 있어서 블로그 초안용으로는 부족할 수 있다. 블로그 한 편이 보통 50~80 블록. 월 10편 쓰면 금방 찬다. 월 $5면 무제한 — 운영 목적이면 유료 권장.

### Notion을 완전히 대체할 수 있나요?

DB·관계형 데이터·협업이 필요하면 불가능하다. Craft에는 DB 기능 자체가 없다. 글쓰기·문서 작성으로 한정하면 대체 가능.

### 둘 다 써도 괜찮나요?

오히려 권장한다. Craft=글쓰기, Notion=프로젝트관리로 역할을 분리하면 각자의 강점을 살린다. 월 $15 이면 둘 다 유료로 쓸 수 있고, ROI 는 충분히 나온다.

### Android 에서 Craft 를 쓸 방법이 있나요?

없다. 웹 뷰어만 존재하고 편집 불가. Android 가 메인이면 Notion 또는 Obsidian이 현실적.

### Craft 의 "Daily Notes" 는 Obsidian 과 비교하면?

Obsidian Daily Notes 가 플러그인 확장성·Dataview 조합 면에서 우위. Craft Daily Notes 는 아름답지만 쿼리가 약하다. 저널링 목적이면 Craft, 데이터화하려면 Obsidian.

## 마무리

Craft 와 Notion 은 "경쟁자" 보다 "도구 동료" 에 가깝다. 글쓰기 경험이 최우선이면 Craft, 데이터 구조화가 핵심이면 Notion. 둘 다 필요한 사람이 대부분이라는 게 현실이고, 그래서 역할 분리가 가장 현실적인 답이다. 이 글은 분기 1회 갱신해 실제 사용 패턴 변화를 반영한다.
